Create and manage markdown notes attached to entities, including daily notes.

## Features

### Automatic Title Extraction

Notes automatically extract their title from the first `# ` (H1) heading in the markdown content. This processing happens asynchronously via the content processor subscription after note events are emitted. The title is displayed in note listings and stored in the database for querying.

You can override automatic extraction with `rei note set-title` to set a stable title that won't change when the note content is edited. This is useful for wikilinks — other notes linking to `[[My Note]]` won't break if you edit the H1 heading. Use `rei note clear-title` to revert to automatic extraction.

### Task Extraction

Notes can contain markdown task lists (`- [ ]` and `- [x]`). Tasks are:
- Automatically discovered when notes are created or updated
- Tracked with content-based hashing for idempotency
- Propagated to their parent entities (intention, action, outcome, or reflection)
- Synchronized bidirectionally (completing a task in the note updates the entity)

## Subcommands

### new

Create a new note.

```bash
rei note new [options]
```

**Options:**
- `-i, --intention <id>` — Attach to intention (optional if fzf is available)
- `-a, --action <id>` — Attach to action
- `-o, --outcome <id>` — Attach to outcome
- `-r, --reflection <id>` — Attach to reflection
- `-h, --habit <id>` — Attach to habit
- `-t, --tag <tag>` — Add tag (repeatable)
- `-c, --category <slug>` — Category to assign (e.g., 'work/meeting')
- `--template <name>` — Template to use for initial content (default: `default`)
- `--actor <name>` — Creator identifier (e.g., "user", "claude-code")
- `--stdin` — Read content from stdin instead of opening editor
- `--archive` — Archive the note immediately after creation. Useful for capturing reference material (web pages, transcripts, raw text) that should be reachable by ID and wikilink but should not appear in default listings. See the [Workflow: capture and summarize](#workflow-capture-and-summarize) example below.

Notes require an anchor entity (intention or habit), and can optionally be attached to a sub-entity (action, outcome, or reflection) when anchored to an intention.

> **FZF Support:** If `-i` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from active intentions.

**Examples:**

```bash
# Create a note for an intention
rei note new -i intention_01h455vb4pex

# Let fzf pick the intention interactively
rei note new

# Create a note for an action with tags
rei note new -i intention_01h455vb4pex -a action_01h455vb4pex -t decision -t architecture

# Create a note for a reflection
rei note new -i intention_01h455vb4pex -r reflection_01h455vb4pex

# Create a note for a habit
rei note new --habit habit_01h455vb4pex

# Create a note with a category
rei note new -i intention_01h455vb4pex --category work/meeting

# Create a note using the daily template
rei note new -i intention_01h455vb4pex --template daily
```

### today

Open or create today's daily note.

```bash
rei note today [options]
```

**Options:**
- `-d, --date <YYYY-MM-DD>` — Date for the note (defaults to today)
- `-t, --tag <tag>` — Add tag (repeatable)
- `--template <name>` — Template to use (default: `daily`)
- `--actor <name>` — Creator identifier (e.g., "user", "claude-code")

Daily notes are anchored to a **JournalEntry** entity — each day gets its own unique ID. This allows notes to be properly linked to a specific day while reusing all the standard note infrastructure.

**Behavior:**
1. Determines the target day (today or from `--date` override)
2. Gets or creates a JournalEntry for that day
3. Checks if a note already exists for that journal entry
4. Injects custom properties as YAML frontmatter (see below)
5. If note exists → opens it in `$EDITOR`
6. If no note exists → creates a new note with frontmatter and opens it in `$EDITOR`
7. On save, emits events for async processing (title extraction, tasks)

**Daily Note Frontmatter:**

Custom properties set on a journal entry (e.g., via `rei day set-property`) are rendered as YAML frontmatter at the top of the daily note file. Frontmatter is regenerated from the event store each time the note is opened, keeping the event store authoritative.

```markdown
---
mood: great
energy: high
sleep-hours: 7.5
---
<!-- Note: note_01abc... -->
```

No frontmatter delimiters are added when no properties are set.

**Examples:**

```bash
# Open or create today's daily note
rei note today

# Create/open note for a specific date
rei note today --date 2026-01-10

# Create today's note with tags
rei note today -t journal -t personal
```

**Use Cases:**
- Daily journaling and planning
- Quick capture of daily thoughts
- Daily standup notes
- Time-boxed reflection

**Daily Note Task Due Dates:**

Tasks added to daily notes automatically get a due date set to that day. For example, if you add `- [ ] Review PR` to your January 15th daily note, that task will have a due date of January 15th.

### list

List notes with their titles.

```bash
rei note list [options]
```

**Options:**
- `-i, --intention <id>` — Filter by intention (includes notes on sub-entities)
- `-a, --action <id>` — Filter by action
- `-o, --outcome <id>` — Filter by outcome
- `-r, --reflection <id>` — Filter by reflection
- `-h, --habit <id>` — Filter by habit
- `-t, --tag <tag>` — Filter by tag
- `-c, --category <slug>` — Filter by category (e.g., 'system/journal')
- `--title <substring>` — Filter by title substring (case-insensitive, matched in SQL)
- `-d, --include-descendants` — Include notes from descendant categories (use with `-c`)
- `--collection <name>` — Filter by collection membership (manual or virtual)
- `-w, --where <filter>` — Filter by custom property value (see Property Filter Syntax below)
- `--include-archived` — Include archived notes in the listing. Archived notes are hidden by default; passing this flag adds them back, with an `[archived]` tag in each line. Composes with all other filters (tag, category, anchor, property, collection).
- `--json` — Output as JSON (for scripting)

> **FZF Support:** If no anchor, tag, category, property, collection, or `--title` filter is provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select an entity to filter by.

**Intention Filter — Smart Aggregation:**

When filtering by intention (`-i`), the command shows **all notes within that intention's scope**, including:
- Notes attached directly to the intention
- Notes attached to actions under that intention
- Notes attached to outcomes under that intention
- Notes attached to reflections under that intention

Notes are grouped by anchor type for readability.

**Examples:**

```bash
# List all notes for an intention (includes notes on actions, outcomes, reflections)
rei note list -i intention_01h455vb4pex

# List notes for a specific action only
rei note list -a action_01h455vb4pex

# List notes for a habit
rei note list --habit habit_01h455vb4pex

# List all notes with a specific tag
rei note list --tag architecture

# Find notes whose title contains "architecture" (case-insensitive, no anchor required)
rei note list --title architecture

# Combine --title with another filter
rei note list -i intention_01h455vb4pex --title review

# Use fzf to select an entity interactively
rei note list

# List notes by category (e.g., daily journal notes)
rei note list --category system/journal

# List notes from a category and all its subcategories
rei note list --category work --include-descendants

# Filter notes by collection membership
rei note list --collection "Reading List"

# Filter by custom property value
rei note list --where "status=reviewed"
rei note list --where "priority:exists"
rei note list --where "priority:missing"
rei note list --where "topics any architecture,design"

# Combine with anchor/tag/category filters
rei note list -i intention_01h455vb4pex --where "status=reviewed"

# Include archived notes (hidden by default); they appear with an [archived] tag
rei note list -i intention_01h455vb4pex --include-archived

# Output as JSON (for scripting)
rei note list --json
```

**Property Filter Syntax:** the same operator set as [`rei intention list --where`](/docs/commands/intention#list):
- `key=value` — Exact match (works with enum, text, and state properties)
- `key>N`, `key>=N`, `key<N`, `key<=N` — Numeric comparison (for int properties)
- `key:exists` / `key:missing` — Presence check
- `key:tag=value` — State machine tag query
- `key any v1,v2,...` — LabelSet / tag-set contains any of the listed values
- `key all v1,v2,...` — LabelSet / tag-set contains all of the listed values
- `category/key=value` — Scope the filter to a specific category's property

Multiple `--where` flags combine with AND semantics. `--where` is applied after the anchor / tag / category filter.

### open

Open a note in your editor, or update it from stdin.

```bash
rei note open [note-id] [--stdin]
```

**Arguments:**
- `[note-id]` — Note ID (optional if fzf is available)

**Options:**
- `--stdin` — Read updated content from stdin instead of opening editor

> **FZF Support:** If note ID is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from existing notes.

**Behavior (default):**
1. Opens the note file in your `$EDITOR`
2. After editing, computes content hash and compares with stored hash
3. If content changed, emits a `NoteContentUpdated` event
4. Title and tasks are re-extracted automatically

**Behavior (`--stdin`):**
1. Reads content from stdin and writes it to the note file
2. Computes content hash and emits a `NoteContentUpdated` event
3. No editor is opened — fully non-interactive

This is useful for programmatic note updates from scripts or agents (e.g., Claude Code) where launching `$EDITOR` is not possible.

**Examples:**

```bash
# Open specific note in editor
rei note open note_01h455vb4pex

# Use fzf to select note interactively
rei note open

# Update note content from stdin (agent-friendly)
echo "# Updated Title\n\nNew content" | rei note open note_01h455vb4pex --stdin

# Pipe a file into an existing note
rei note open note_01h455vb4pex --stdin < path/to/updated-content.md
```

### get-help

Get AI help with a note. This invokes Claude with context about the note, including its content, anchor entity, and any configured guidance.

```bash
rei note get-help [note-id] [options]
```

**Arguments:**
- `[note-id]` — Note ID (optional if fzf is available)

**Options:**
- `-f, --focus <text>` — Focus area to emphasize (e.g., 'how to break this down')
- `--no-guidance` — Skip intention and category-level note guidance
- `--include-doc` — Include content from attached documents in context
- `-i, --interactive` — Start an interactive Claude Code session
- `--debug` — Output the rendered prompt without invoking the LLM
- `--append` — Append help to the original note instead of creating a new note

> **FZF Support:** If note ID is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from existing notes.

**Behavior:**

When the agent creates a help note, it automatically includes a wikilink back to the original note (if the original has a title). This creates a navigable relationship between the notes, and the link will appear in the original note's backlinks.

**Guidance Sources:**

Guidance can come from two places, and both are included by default:
1. **Intention guidance** — Set via `intention set-note-guidance` on the anchor intention
2. **Category guidance** — Set via `category set-note-guidance` on the note's assigned category

Category guidance provides domain knowledge (e.g., "how to study kanji") while intention guidance provides user-specific context. Use `--no-guidance` to skip both.

The `--focus` flag lets you emphasize a specific aspect or question without replacing the guidance context.

**Interactive Mode:**

Use `-i` or `--interactive` to start an interactive Claude Code session instead of a one-shot response. This allows back-and-forth conversation about the note.

**Document Context:**

Use `--include-doc` to include the content of attached documents in the agent context. Text-based files (md, txt, json, yaml, etc.) have their content read; binary files are noted but content is not included.

**Debug Mode:**

Use `--debug` to see the full prompt that would be sent to the LLM without actually invoking it:

```
=== DEBUG: Rendered prompt that would be sent to agent ===
[Full prompt content]
=== END DEBUG ===
```

**Append Mode:**

Use `--append` to add the AI help directly to the original note instead of creating a separate help note. This appends a "## AI Help" section with a horizontal rule separator.

**Examples:**

```bash
# Get help with a specific note
rei note get-help note_01h455vb4pex

# Use fzf to select note
rei note get-help

# Get help with a specific focus
rei note get-help note_01h455vb4pex --focus "how to structure this document"

# Skip all guidance
rei note get-help note_01h455vb4pex --no-guidance

# Include attached documents in context
rei note get-help note_01h455vb4pex --include-doc

# Start an interactive session
rei note get-help note_01h455vb4pex -i

# Debug: see the prompt without invoking LLM
rei note get-help note_01h455vb4pex --debug

# Append help to the original note
rei note get-help note_01h455vb4pex --append
```

### move

Move a note to a different entity (change its anchor).

```bash
rei note move [-n NOTE_ID] [anchor-options]
```

**Options:**
- `-n, --note <id>` — Note ID to move (optional if fzf is available)
- `-i, --intention <id>` — Move to intention
- `-a, --action <id>` — Move to action
- `-o, --outcome <id>` — Move to outcome
- `-r, --reflection <id>` — Move to reflection
- `-h, --habit <id>` — Move to habit
- `--disruption <id>` — Move to disruption
- `--disruption-action <id>` — Move to disruption action

> **FZF Support:** If `-n` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from existing notes. If no anchor flag is provided, a toggle picker lets you select the target entity.

**Behavior:**
- Emits a single atomic `NoteAnchorChanged` event recording both the old and new anchor
- Idempotent: moving to the same anchor is a no-op
- Deleted notes cannot be moved
- Pre-existing tasks stay attributed to the original entity (historical accuracy); future tasks use the new anchor

**Examples:**

```bash
# Move a note to a different intention
rei note move -n note_01h455vb4pex -i intention_01h999abc

# Use fzf to select both note and target
rei note move

# Move a note to a habit
rei note move -n note_01h455vb4pex --habit habit_01h888xyz

# Move a note to an action under an intention
rei note move -n note_01h455vb4pex -i intention_01h999abc -a action_01h777def
```

**Options:**
- `--path <dir>`, `-p <dir>` — Destination directory for the exported `.md` file. Required for the file to actually be written; optional at the parser layer so the FZF picker can run first.

> **FZF Support:** If the note ID is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker launches to let you select from every note in the workspace. This happens whether or not `--path` is supplied, so running bare `rei note export` still shows the picker — but if you then have no `--path` set, the command prints an error reminding you to supply it and exits.

**Behavior:**
1. Resolves the note (argument or FZF selection).
2. If `--path` was not supplied, prints a usage reminder and exits.
3. Looks up the note's title (used to derive the destination filename).
4. Copies `<workspaceRoot>/notes/<noteId>.md` into `<dir>/<slug-of-title>.md` (or `<dir>/<noteId>.md` when the note has no title).
5. Creates `<dir>` if it does not exist. Overwrites existing files silently.
6. Emits no events — this is a read-only, side-effecting operation.

**Examples:**

```bash
# Export a specific note to a directory
rei note export note_01h455vb4pex --path ~/Desktop/exports

# Use fzf to select the note, relative path
rei note export --path ./exported-notes

# Bare: pick the note interactively, then get a reminder about --path
rei note export

# Tilde expansion works
rei note export note_01h455vb4pex --path '~/share/onboarding'
```

### delete

Delete a note from the workspace.

```bash
rei note delete [note-id]
```

**Arguments:**
- `[note-id]` — Note ID (optional if fzf is available)

> **FZF Support:** If note ID is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from existing notes.

**Behavior:**
1. Removes the note file from the workspace
2. Emits a `NoteDeleted` event
3. Git subscription handler commits the file removal (if git integration enabled)

**Examples:**

```bash
# Delete a specific note
rei note delete note_01h455vb4pex

# Use fzf to select note
rei note delete
```

**Warning:** This action is permanent. The note file is removed from disk.

### archive

Move a note out of the default listings without deleting it. Archived notes are hidden by `rei note list` and cross-module surfaces (intention show, today/yesterday, day review, habit show, disruption show, FZF preview, agent context builders) but remain reachable by ID, can be opened, edited, and referenced by wikilinks. Use [`unarchive`](#unarchive) to restore.

```bash
rei note archive [note-id]
```

**Arguments:**
- `[note-id]` — Note ID (optional if fzf is available)

> **FZF Support:** If note ID is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker opens with **active** notes only. To archive a note that is already archived, pass its ID directly; the picker does not offer archived notes.

**Behavior:**
1. Emits a `NoteArchived` event on the note's stream.
2. The read-model row's `status` flips to `archived` and `archived_at` is set.
3. The `.md` file stays put on disk — archive is a metadata transition, not a filesystem move.
4. Idempotent: archiving an already-archived note is a no-op.
5. Cannot archive a deleted note (returns an error).

**Examples:**

```bash
# Archive a specific note
rei note archive note_01abc123

# Use fzf to pick from active notes
rei note archive
```

**Example output:**

```
Archived note "Captured: example.com" (note_01abc123)
```

### unarchive

Reverse `archive`. The note rejoins the default listings.

```bash
rei note unarchive [note-id]
```

**Arguments:**
- `[note-id]` — Note ID (optional if fzf is available, but see note below)

> **FZF Support:** If note ID is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker opens — but it shows only **active** notes, which means it is *not* useful for unarchive in practice. To find an archived note's ID, run `rei note list --include-archived` first, then pass the ID explicitly.

**Behavior:**
1. Emits a `NoteUnarchived` event on the note's stream.
2. The read-model row's `status` flips to `active` and `archived_at` is cleared.
3. Idempotent: unarchiving an active note is a no-op.
4. Cannot unarchive a deleted note (returns an error).

**Examples:**

```bash
# Find archived candidates first
rei note list -i intention_01h455vb4pex --include-archived

# Unarchive by ID
rei note unarchive note_01abc123
```

**Example output:**

```
Unarchived note "Captured: example.com" (note_01abc123)
```

## File Embeds

Notes support embedding files using Obsidian-style syntax:

```markdown
See the diagram: ![[architecture.png]]

Reference: ![[research-paper.pdf]]
```

**Syntax:**
- `![[filename]]` — Embed a file from the workspace

**Processing:**
- Embeds are discovered when notes are created or updated
- The file path is resolved relative to the workspace docs directory
- Embed events are propagated to the anchor entity's stream
- Use `rei knowledge verify --embeds` to check for missing embedded files

## Wikilinks

Notes support wikilink syntax for linking between notes:

```markdown
See [[Project Notes]] for more details.

Check [[Meeting Notes|the meeting notes]] from last week.
```

**Syntax:**
- `[[Note Title]]` — Link to a note by title
- `[[Note Title|Display Text]]` — Link with custom display text

**Processing:**
- Wikilinks are discovered when notes are created or updated
- References are automatically resolved when a matching note title exists
- If a note is deleted or renamed, references may become "broken"

**Resolution:**
- When a note's title is extracted, any unresolved references matching that title are automatically resolved
- Use `rei note broken-links` to find unresolved references

## Workspace

Notes are stored as markdown files in your workspace directory (default: `.rei/notes/`).

You can edit them directly with any text editor or use `rei note open`.

## Workflow: capture and summarize

A common pattern: capture a webpage, transcript, or document as raw markdown for reference, then write a clean summary that links back to the raw capture. The capture would otherwise pollute default listings; the summary is what you actually want to see day-to-day. Archive the capture so only the summary surfaces by default; the wikilink from the summary to the capture continues to resolve.

Steps:

```bash
# 1. Capture the page as a raw archived note.
rei note new -i intention_01h455vb4pex --archive
# editor opens; paste the markdown rendering of the page; save the
# title as e.g. "Wikipedia: Foo (raw)".

# 2. Write a summary that wikilinks to the capture.
rei note new -i intention_01h455vb4pex
# editor opens; write the summary using [[Wikipedia: Foo (raw)]] as
# the wikilink target.

# 3. List notes — only the summary appears.
rei note list -i intention_01h455vb4pex
#   note_01summary
#     Title: Foo summary
#     ...

# 4. The summary's outgoing links still resolve to the archived
#    capture (the wikilink target is identified by title, not
#    status). The default listing hides archived targets:
rei note outgoing-links note_01summary

# 5. To see the archived target in the listing too, opt in:
rei note outgoing-links note_01summary --include-archived

# 6. To revisit the captured page directly, fetch by ID:
rei note show note_01raw     # works without --include-archived
rei note open note_01raw
```

This workflow is the primary motivation for the `--archive` flag on `note new` — it captures and hides in one shot, so you don't have to remember a follow-up `rei note archive`. The captured note remains a first-class, editable, wikilink-resolvable note; it is only excluded from default listings and dashboards.
