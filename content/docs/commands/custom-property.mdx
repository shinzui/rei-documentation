Define and manage custom properties that can be attached to entities (intentions, habits, disruptions, notes, journal entries, collections, links, actions, outcomes, blockers). Custom properties allow you to extend entities with your own metadata fields.

## Concepts

**Custom Property** — A user-defined property definition with:
- **Key** — Unique identifier (alphanumeric, hyphens, underscores)
- **Label** — Human-readable display name
- **Value type** — The data type for values (int, enum, state-machine, etc.)
- **Entity scope** — Which entity types can have this property
- **Category scope** — Optional restriction to a specific category

**Property Assignment** — Once a property is defined, you can assign values to entities using `rei <entity> set-property` commands.

## Value Types

Custom properties support the following value types:

| Type | Description | Example |
|------|-------------|---------|
| `int` | Integer with optional min/max | effort level (1-10) |
| `float` | Decimal with optional precision | completion percentage |
| `text` | Free text with optional length limits | notes field |
| `bool` | Boolean true/false | is_urgent flag |
| `enum` | Predefined set of values (single-select) | priority (low, medium, high) |
| `label-set` | Predefined set of labels (multi-select) | areas (backend, frontend, security) |
| `duration` | Time duration | estimated time |
| `quantity` | Numeric with units | distance (km, miles) |
| `state-machine` | States with transitions | task workflow |
| `date` | Date (YYYY-MM-DD) | due date |
| `datetime` | Date and time | deadline |
| `path` | File path | reference file |
| `url` | URL | documentation link |
| `note` | Reference to a note entity | spec note, next action note |
| `git-ref` | Git commit SHA or SHA range | deployed commit, change range |
| `uuid` | UUID (optionally version-constrained) | external ticket ID |
| `tag-set` | Free-form set of tags (no predeclared values) | topics, keywords |
| `path-list` | Unique, sorted set of file paths | related specs, source paths |

## State Machine YAML Format

State machine properties require a YAML configuration file that defines states and transitions. The file is passed via `--from-file` when creating the property.

**Basic Structure:**

```yaml
initialState: backlog        # Required: state to assign when no value exists
states:                      # Required: at least one state
  - stateId: backlog
    stateLabel: "Backlog"
    stateDescription: "Items waiting to be picked up"  # Optional
    stateTags:               # Optional: semantic tags for querying
      - systemTag: TagQueue
    stateColor: "#808080"    # Optional: hex color for UI
  - stateId: in-progress
    stateLabel: "In Progress"
    stateTags:
      - systemTag: TagActive
  - stateId: done
    stateLabel: "Done"
    stateTags:
      - systemTag: TagDone
      - systemTag: TagTerminal
transitions:                 # Required: defines allowed state changes
  - transitionFrom: backlog
    transitionTo: in-progress
    transitionLabel: "Start"  # Optional
  - transitionFrom: in-progress
    transitionTo: done
    transitionLabel: "Complete"
  - transitionFrom: in-progress
    transitionTo: backlog
    transitionLabel: "Move back"
```

**System Tags:**

System tags provide semantic meaning for queries and UI:

| Tag | Description |
|-----|-------------|
| `TagQueue` | Items waiting to be worked on |
| `TagActive` | Items currently in progress |
| `TagDone` | Items that are complete |
| `TagTerminal` | Final states (no further transitions) |

**Custom Tags:**

You can also define custom tags for your own categorization:

```yaml
stateTags:
  - userTag: "needs-review"
  - userTag: "blocked-external"
```

**Example: Kanban Workflow**

```yaml
# kanban.yaml
initialState: backlog
states:
  - stateId: backlog
    stateLabel: "Backlog"
    stateTags:
      - systemTag: TagQueue
  - stateId: ready
    stateLabel: "Ready"
    stateTags:
      - systemTag: TagQueue
  - stateId: in-progress
    stateLabel: "In Progress"
    stateTags:
      - systemTag: TagActive
  - stateId: review
    stateLabel: "In Review"
    stateTags:
      - systemTag: TagActive
      - userTag: "needs-review"
  - stateId: done
    stateLabel: "Done"
    stateTags:
      - systemTag: TagDone
      - systemTag: TagTerminal
  - stateId: cancelled
    stateLabel: "Cancelled"
    stateTags:
      - systemTag: TagTerminal
transitions:
  - transitionFrom: backlog
    transitionTo: ready
    transitionLabel: "Groom"
  - transitionFrom: ready
    transitionTo: in-progress
    transitionLabel: "Start"
  - transitionFrom: in-progress
    transitionTo: review
    transitionLabel: "Submit for review"
  - transitionFrom: review
    transitionTo: in-progress
    transitionLabel: "Request changes"
  - transitionFrom: review
    transitionTo: done
    transitionLabel: "Approve"
  - transitionFrom: backlog
    transitionTo: cancelled
    transitionLabel: "Cancel"
  - transitionFrom: ready
    transitionTo: cancelled
    transitionLabel: "Cancel"
  - transitionFrom: in-progress
    transitionTo: cancelled
    transitionLabel: "Cancel"
```

Create the property with:

```bash
rei custom-property create workflow \
  --type-state-machine --from-file kanban.yaml \
  -e intention \
  --label "Workflow Status"
```

## Subcommands

### create

Create a new custom property definition.

```bash
rei custom-property create KEY [options] --type-<TYPE> [type-options]
```

**Arguments:**
- `KEY` — Property key (alphanumeric, hyphens, underscores)

**Options:**
- `-l, --label <label>` — Human-readable label (defaults to key)
- `-d, --description <desc>` — Property description
- `-e, --entity-type <type>` — Entity type to apply to (can be repeated). Valid types: intention, habit, disruption, note, journal_entry, collection, link
- `-c, --category <slug>` — Scope to a specific category (must be a leaf category with no children)
- `--inheritable` — Mark the property as inheritable. When set on an intention, descendants in the intention tree see the value (nearest ancestor wins) unless they set their own local value. State-machine value types cannot be inheritable. See [Inheritance in the Intention Tree](#inheritance-in-the-intention-tree).
- `--actor <actor>` — Actor performing this action (overrides global `--actor`)

**Type Flags (pick one):**

- `--type-int` — Integer type
  - `--min <N>` — Minimum value
  - `--max <N>` — Maximum value

- `--type-float` — Float type
  - `--min <N>` — Minimum value
  - `--max <N>` — Maximum value
  - `--precision <N>` — Decimal precision

- `--type-text` — Text type
  - `--min-length <N>` — Minimum length
  - `--max-length <N>` — Maximum length

- `--type-bool` — Boolean type

- `--type-enum` — Enum type (single-select)
  - `--values <val1,val2,...>` — Comma-separated enum values (required)

- `--type-label-set` — Label set type (multi-select)
  - `--labels <label1,label2,...>` — Comma-separated labels (required)

- `--type-duration` — Duration type

- `--type-quantity` — Quantity type (numeric with units)
  - `--base-unit <unit>` — Base storage unit (required)
  - `--allowed-units <u1,u2,...>` — Comma-separated allowed units (required)
  - `--display-unit <unit>` — Preferred display unit

- `--type-state-machine` — State machine type
  - `--from-file <file>` — YAML file defining states and transitions (required)

- `--type-date` — Date type (YYYY-MM-DD)

- `--type-datetime` — DateTime type

- `--type-path` — File path type
  - `--must-exist` — Path must exist at assignment time
  - `--must-be-directory` — Path must be a directory
  - `--must-be-file` — Path must be a file
  - `--must-be-absolute` — Path must be absolute

- `--type-path-list` — Unique set of file paths (comma-separated at assignment)
  - Same constraint flags as `--type-path` (`--must-exist`, `--must-be-directory`, `--must-be-file`, `--must-be-absolute`), applied to every path in the set.
  - Duplicate paths in the input are silently collapsed via set semantics.
  - Values are stored sorted and rendered joined by `", "`.
  - `--where "key any path1,path2"` and `--where "key all path1,path2"` work as for tag sets.

- `--type-url` — URL type
  - `--allowed-schemes <s1,s2,...>` — Comma-separated allowed schemes
  - `--allowed-hosts <h1,h2,...>` — Comma-separated allowed hosts

- `--type-note` — Note reference type (value is a note ID)

- `--type-git-ref` — Git commit reference type
  - `--ref-format <format>` — Allowed format: `commit` (single SHA), `range` (SHA..SHA or SHA...SHA), or `any` (default)

- `--type-uuid` — UUID type
  - `--uuid-version <version>` — Allowed UUID version: `v4`, `v7`, or `any` (default: `any`)

- `--type-tag-set` — Free-form tag set (no predeclared values)
  - No options. Tags are provided at assignment time as a comma-separated list and validated per-tag (must be non-empty after trim, no newlines). Unlike `--type-label-set`, there is no allowed-values list on the definition, so no `add-label` / `remove-label` evolution commands apply.

**Examples:**

```bash
# Create a priority enum property for intentions
rei custom-property create priority \
  --type-enum --values low,medium,high \
  -e intention

# Create an effort int property for intentions and habits
rei custom-property create effort \
  --type-int --min 1 --max 10 \
  -e intention -e habit \
  --label "Effort Level" \
  --description "Estimated effort from 1 (trivial) to 10 (massive)"

# Create a severity enum for disruptions
rei custom-property create severity \
  --type-enum --values low,medium,high,critical \
  -e disruption

# Create a label-set property for tagging areas (multi-select)
rei custom-property create areas \
  --type-label-set --labels backend,frontend,security,devops \
  -e intention \
  --label "Work Areas"

# Create a state machine property from YAML config
rei custom-property create task-state \
  --type-state-machine --from-file task-states.yaml \
  -e intention

# Create a note reference property for cross-referencing
rei custom-property create spec \
  --type-note \
  -e intention

# Create a git ref property for tracking deployed commits
rei custom-property create deployed-commit \
  --type-git-ref --ref-format commit \
  -e intention

# Create a git ref property that accepts commit ranges
rei custom-property create change-range \
  --type-git-ref --ref-format range \
  -e intention

# Create a text property scoped to a category
rei custom-property create client-notes \
  --type-text --max-length 1000 \
  -e intention \
  -c work

# Create a free-form tag set property for topics on notes
rei custom-property create topics \
  --type-tag-set \
  -e note \
  --label "Topics"

# Assign tags at value-set time (comma-separated)
rei note set-property NOTE_ID topics "haskell,event-sourcing,decider"

# Filter by any/all using the same operators as label-set
rei note list --where "topics any haskell,rust"
```

**Notes:**
- Category scope (`-c`) requires a leaf category (one with no children). This ensures properties are assigned at the most specific level.

### list

List custom property definitions.

```bash
rei custom-property list [options]
```

**Options:**
- `-a, --all` — Include archived properties
- `-c, --category <slug>` — Filter by category scope
- `-e, --entity-type <type>` — Filter by entity type

**Examples:**

```bash
# List all active properties
rei custom-property list

# List properties for intentions only
rei custom-property list -e intention

# Include archived properties
rei custom-property list --all

# Filter by category scope
rei custom-property list -c work
```

### show

Show details of a custom property.

```bash
rei custom-property show [KEY] [options]
```

**Arguments:**
- `[KEY]` — Property key or ID (optional if fzf is available)

**Options:**
- `--with-counts` — Include entity counts per state (state machine properties only)
- `-c, --category <slug>` — Category scope for disambiguation when multiple properties have the same key
- `--json` — Output as JSON

> **FZF Support:** If the key is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from available properties.

**Examples:**

```bash
# Show property details
rei custom-property show priority

# Use FZF to select
rei custom-property show

# Show state machine property with entity counts per state
rei custom-property show workflow --with-counts

# Show category-scoped property (when key exists in multiple categories)
rei custom-property show workflow --category work
```

**Notes:**
- The `--with-counts` flag shows the number of entities currently in each state, useful for tracking WIP limits

### entities

List all entities that have a custom property set.

```bash
rei custom-property entities [KEY] [options]
```

**Arguments:**
- `KEY` — Property key or ID (optional; launches FZF picker if omitted)

**Options:**
- `-v, --value <value>` — Filter to entities with this specific value
- `-e, --entity-type <type>` — Filter by entity type (intention, habit, disruption, note, collection, link, action, outcome, blocker)
- `-c, --category <slug>` — Category scope for disambiguation when multiple properties have the same key
- `--json` — Output as JSON

**Examples:**

```bash
# List all entities with the "priority" property set
rei custom-property entities priority

# Filter to a specific value
rei custom-property entities priority --value high

# Show only intentions
rei custom-property entities workflow --entity-type intention

# Show only actions with the property set
rei custom-property entities workflow --entity-type action

# Disambiguate category-scoped property
rei custom-property entities workflow --category work

# JSON output for scripting
rei custom-property entities priority --json
```

**Notes:**
- Only active/non-archived entities are shown (active/future intentions, active habits, unresolved disruptions, etc.)
- For state machine properties, entities are grouped by state and sorted by state order
- For other property types, entities are shown in a flat list sorted by entity type and name
- Only entity types in the property's allowed scope are queried

### archive

Archive a custom property (soft delete).

```bash
rei custom-property archive KEY [options]
```

**Arguments:**
- `KEY` — Property key or ID

**Options:**
- `-c, --category <slug>` — Category scope for disambiguation when multiple properties have the same key

**Examples:**

```bash
# Archive a property
rei custom-property archive old-priority

# Archive a category-scoped property (when key exists in multiple categories)
rei custom-property archive workflow --category work
```

**Notes:**
- Archived properties are hidden from lists by default
- Existing property assignments are preserved
- Use `list --all` to see archived properties

### relabel

Update the display label of a custom property.

```bash
rei custom-property relabel KEY NEW_LABEL [options]
```

**Arguments:**
- `KEY` — Property key or ID
- `NEW_LABEL` — New display label

**Options:**
- `-c, --category <slug>` — Category scope for disambiguation when multiple properties have the same key

**Examples:**

```bash
# Relabel a property
rei custom-property relabel priority "Priority Level"

# Relabel a category-scoped property (when key exists in multiple categories)
rei custom-property relabel workflow "Work Workflow" --category work
```

**Notes:**
- Property keys are immutable; only the label can be changed
- Use labels for human-readable display names

### set-inheritable / unset-inheritable

Toggle the `isInheritable` flag on an existing property.

```bash
rei custom-property set-inheritable KEY [--lookup-category SLUG] [--actor ACTOR]
rei custom-property unset-inheritable KEY [--lookup-category SLUG] [--actor ACTOR]
```

**Arguments:**
- `KEY` — Property key or `custom_property_…` KindID

**Options:**
- `--lookup-category <slug>` — Disambiguate when multiple properties share the same key
- `--actor <actor>` — Actor performing this action (overrides global `--actor`)

**Examples:**

```bash
# Let descendants inherit focus-area values from ancestor intentions
rei custom-property set-inheritable focus-area

# Stop propagating values (local values are preserved; only propagation stops)
rei custom-property unset-inheritable focus-area
```

**Notes:**
- State-machine value types cannot be inheritable; the command is rejected with `Error: State-machine properties cannot be inheritable`.
- Toggling the flag takes effect immediately on subsequent `show`/list reads — there is no re-assignment required.
- See [Inheritance in the Intention Tree](#inheritance-in-the-intention-tree) for full semantics.

### set-scope

Update property scope (entity types and category).

```bash
rei custom-property set-scope KEY_OR_ID [options]
```

**Arguments:**
- `KEY_OR_ID` — Property key or `custom_property_…` KindID

**Options:**
- `-e, --entity-type <type>` — Entity type (can be repeated to set multiple)
- `-c, --category <slug>` — New category scope to set
- `--clear-category` — Remove category scope (make global)
- `--lookup-category <slug>` — Disambiguate property lookup when multiple properties share the same key (has no effect when a KindID is given)

**Examples:**

```bash
# Add habit to entity types
rei custom-property set-scope priority -e intention -e habit

# Scope to a specific category
rei custom-property set-scope client-notes -c work

# Remove category scope
rei custom-property set-scope client-notes --clear-category

# Disambiguate when key exists in multiple categories
rei custom-property set-scope workflow -c personal --lookup-category work
```

## State Machine Evolution

State machine properties can be evolved after creation using these commands.

> **Tip:** All commands in this section accept a `custom_property_…` KindID in place of the key, and `--lookup-category SLUG` for disambiguation when multiple properties share the same key.

### add-state

Add a state to a state machine property.

```bash
rei custom-property add-state KEY STATE_ID --label LABEL [options]
```

**Arguments:**
- `KEY` — Property key or ID
- `STATE_ID` — State identifier (lowercase, hyphens/underscores)

**Options:**
- `-l, --label <label>` — Human-readable state label (required)
- `-d, --description <desc>` — State description
- `-t, --tag <tag>` — State tag (can be repeated). Common tags: queue, active, done, terminal
- `-o, --order <N>` — Display order (lower numbers shown first)

**Examples:**

```bash
# Add a new state
rei custom-property add-state task-state blocked --label "Blocked" -t active

# Add a terminal state
rei custom-property add-state task-state cancelled --label "Cancelled" -t terminal

# Add a state with display order
rei custom-property add-state task-state review --label "In Review" -t active --order 2
```

### remove-state

Remove a state from a state machine property.

```bash
rei custom-property remove-state KEY STATE_ID
```

**Arguments:**
- `KEY` — Property key or ID
- `STATE_ID` — State identifier to remove

**Examples:**

```bash
# Remove a state
rei custom-property remove-state task-state blocked
```

**Notes:**
- Cannot remove states that are currently in use by entities
- Transitions involving the state are also removed

### update-state-order

Update the display order of a state in a state machine property.

```bash
rei custom-property update-state-order KEY STATE_ID [options]
```

**Arguments:**
- `KEY` — Property key or ID
- `STATE_ID` — State identifier to update

**Options:**
- `-o, --order <N>` — Display order (lower numbers shown first). Omit to clear the order.

**Examples:**

```bash
# Set display order for a state
rei custom-property update-state-order task-state in-progress --order 1

# Clear display order (state will sort by ID)
rei custom-property update-state-order task-state in-progress
```

**Notes:**
- States with display order are shown first (sorted by order)
- States without display order are shown after (sorted alphabetically by ID)
- Display order affects CLI output and grouped intention lists

### add-transition

Add a transition between states.

```bash
rei custom-property add-transition KEY FROM_STATE TO_STATE [options]
```

**Arguments:**
- `KEY` — Property key or ID
- `FROM_STATE` — Source state ID
- `TO_STATE` — Target state ID

**Options:**
- `-l, --label <label>` — Transition label

**Examples:**

```bash
# Add a transition
rei custom-property add-transition task-state todo in-progress --label "Start"

# Add reverse transition
rei custom-property add-transition task-state in-progress todo --label "Revert"
```

### remove-transition

Remove a transition between states.

```bash
rei custom-property remove-transition KEY FROM_STATE TO_STATE
```

**Arguments:**
- `KEY` — Property key or ID
- `FROM_STATE` — Source state ID
- `TO_STATE` — Target state ID

**Examples:**

```bash
# Remove a transition
rei custom-property remove-transition task-state todo blocked
```

### set-state-limit

Set a WIP (Work In Progress) limit on a state machine state.

```bash
rei custom-property set-state-limit KEY --state STATE_ID --limit N
```

**Arguments:**
- `KEY` — Property key or ID

**Options:**
- `-s, --state <state_id>` — State identifier to set limit on (required)
- `-l, --limit <N>` — Maximum entities allowed in this state (required)

**Examples:**

```bash
# Set a WIP limit of 3 on the in-progress state
rei custom-property set-state-limit workflow --state in-progress --limit 3

# Set a limit on a category-scoped property
rei custom-property set-state-limit workflow --state in-progress --limit 5 --category work
```

**Notes:**
- WIP limits are advisory — they show warnings but don't block transitions
- Use `--with-counts` on `show` command to see current entity counts per state

### clear-state-limit

Remove a WIP limit from a state machine state.

```bash
rei custom-property clear-state-limit KEY --state STATE_ID
```

**Arguments:**
- `KEY` — Property key or ID

**Options:**
- `-s, --state <state_id>` — State identifier to clear limit from (required)

**Examples:**

```bash
# Remove the WIP limit from a state
rei custom-property clear-state-limit workflow --state in-progress
```

### metrics

View state machine transition metrics and workflow analytics.

```bash
rei custom-property metrics [KEY] [options]
```

**Arguments:**
- `[KEY]` — Property key or ID (state machine properties only, optional if fzf is available)

**Options:**
- `-d, --detailed` — Show detailed breakdown by entity
- `-c, --category <slug>` — Category scope for disambiguation when multiple properties have the same key

**Examples:**

```bash
# View metrics for a state machine property
rei custom-property metrics workflow

# View detailed metrics with entity breakdown
rei custom-property metrics workflow --detailed

# View metrics for category-scoped property
rei custom-property metrics workflow --category work
```

**Notes:**
- Metrics show current state distribution with entity counts
- Average time in each state
- Recent transitions (last 7 days)

## Lifecycle Property

State machine properties can be designated as "lifecycle properties" — when their value enters a terminal state, the system schedules automatic completion of the intention after a configurable delay.

### set-lifecycle

Designate a state machine property as the lifecycle property for its scope.

```bash
rei custom-property set-lifecycle KEY --days N [options]
```

**Arguments:**
- `KEY` — Property key or ID (must be a state machine with terminal states)

**Options:**
- `--days <N>` — Days to wait before auto-completing (must be >= 1, required)
- `-c, --category <slug>` — Category scope for disambiguation

**Examples:**

```bash
# Set lifecycle with 3-day delay
rei custom-property set-lifecycle workflow --days 3

# Set lifecycle on a category-scoped property
rei custom-property set-lifecycle task-status --days 5 --category work
```

**Notes:**
- Only one lifecycle property is allowed per scope (global or per-category)
- The property must have at least one state tagged as `terminal`
- When an intention enters a terminal state, a reminder is scheduled N days out
- If the intention is still in a terminal state when the reminder fires, it is auto-completed
- If the state changes back before the reminder fires, the reminder is cancelled

### revoke-lifecycle

Remove lifecycle designation from a state machine property.

```bash
rei custom-property revoke-lifecycle KEY [options]
```

**Arguments:**
- `KEY` — Property key or ID

**Options:**
- `-c, --category <slug>` — Category scope for disambiguation

**Examples:**

```bash
# Revoke lifecycle designation
rei custom-property revoke-lifecycle workflow
```

## Using Properties

Once properties are defined, assign values to entities:

```bash
# Set a property on an intention
rei intention set-property intention_01abc... priority high

# Set a label-set property (multiple labels, comma-separated)
rei intention set-property intention_01abc... areas backend,security

# Set a property on a habit
rei habit set-property habit_01abc... effort 5

# Set a property on a disruption
rei disruption set-property redis-timeouts severity critical

# Set a property on a note
rei note set-property note_01abc... topics haskell,event-sourcing

# Set a property on a collection
rei collection set-property collection_01abc... status active

# Set a property on a link
rei link set-property link_01abc... reading-status reading

# Set a property on a journal entry (today's day)
rei day set-property mood great

# Set a property on a specific date
rei day set-property --date 2026-02-12 mood good

# Clear a property from any entity
rei intention clear-property intention_01abc... priority
rei habit clear-property habit_01abc... effort
rei disruption clear-property redis-timeouts severity
rei note clear-property note_01abc... topics
rei collection clear-property collection_01abc... status
rei link clear-property link_01abc... reading-status
rei day clear-property mood
```

### Forcing State Machine Transitions

State machine properties normally require following defined transitions (e.g., `backlog` → `in-progress` → `review` → `done`). If you need to skip intermediate states, use `--force`:

```bash
# Skip from backlog directly to done (bypasses transition rules)
rei intention set-property intention_01abc... workflow done --force

# Also works on habits
rei habit set-property habit_01abc... workflow done --force
```

The `--force` flag bypasses transition rule validation but still enforces terminal state restrictions — you cannot leave a terminal state even with `--force`. The `--force` flag has no effect on non-state-machine properties.

## Filtering by Properties

Use the `--where` flag to filter entity lists by property values:

```bash
# Filter intentions by priority
rei intention list --where priority=high

# Filter by numeric comparison
rei intention list --where "effort>=5"

# Filter habits by effort
rei habit list --where "effort<3"
```

### LabelSet / TagSet Filtering

Label-set and tag-set properties support two special operators for filtering:

- `any` — Match entities that have **any** of the specified values
- `all` — Match entities that have **all** of the specified values

```bash
# Find intentions with any of the specified areas
rei intention list --where "areas any backend,frontend"

# Find intentions that have all specified areas
rei intention list --where "areas all backend,security"

# Works with disruptions too
rei disruption list --where "impact-areas any infrastructure,api"
```

See [`rei intention list`](/docs/commands/intention), [`rei habit list`](/docs/commands/habit), and [`rei disruption list`](/docs/commands/disruption) for more filtering options.

## Automatic Initialization

### init-entities

Retroactively initialize a state machine property for entities that have the matching category but don't have a value set. The backfill loop covers categorized intentions, habits, disruptions, notes, collections, links, actions, outcomes, and blockers.

```bash
rei custom-property init-entities KEY_OR_ID [options]
```

**Arguments:**
- `KEY_OR_ID` — Property key or `custom_property_…` KindID

**Options:**
- `-n, --dry-run` — Show what would be done without making changes
- `--lookup-category <slug>` — Disambiguate property lookup

**Examples:**

```bash
# Preview what would be initialized
rei custom-property init-entities task-state --dry-run

# Initialize all matching entities
rei custom-property init-entities task-state
```

**Notes:**
- Only works with state machine properties that are scoped to a category
- Only initializes entities that belong to the property's scoped category
- Skips entities that already have a value for this property
- Sets the property to the state machine's `initialState`
- Includes actions, outcomes, and blockers whose category carries the state machine property

**Automatic Initialization:**

State machine properties scoped to a category are automatically initialized when:
1. A category is assigned to an entity via `rei intention set-category` (or similar)
2. The category has state machine properties scoped to it
3. The entity doesn't already have a value for those properties

The `init-entities` command is useful for retroactively initializing existing entities that were created before the property was defined.

## Property Resolution and Inheritance

When you assign a property to an entity (e.g., `rei intention set-property ... priority high`), the system needs to decide *which* property definition applies. Resolution depends on the entity's **resolution category** and the property's scope.

### Resolution Category by Entity Type

| Entity | Resolution category used |
|--------|---------------------------|
| Intention | Own category if set; otherwise walks up the **parent intention chain** to the nearest ancestor with a category |
| Habit | Own category only |
| Note | Own category only (does **not** inherit from the entity the note is attached to) |
| Disruption | Own category only |
| Collection | Own category only |
| Link | Own category only |
| Journal entry | None — only global properties apply |

### Intention Parent Inheritance (Category Context)

Intentions are the only entity type that inherits a resolution category from an ancestor. If a child intention has no category of its own, the system walks up its parent chain until it finds an ancestor with a category, and uses that category for resolution.

This means a category-scoped property defined on `work` is available to a child intention whose parent (or grandparent, etc.) is categorized as `work`, even if the child itself has no category set.

**Important distinction:** what is inherited by default is the *category context* used to decide which property definition applies — **not** property values. Setting `priority=high` on a parent intention does NOT propagate that value to its children unless the property definition is marked `--inheritable`. See [Inheritance in the Intention Tree](#inheritance-in-the-intention-tree) for how to opt in to value inheritance.

### Resolution Precedence

When multiple property definitions could match a given key on an entity, the system picks the most specific one:

1. **Category-scoped beats global.** If a `priority` property exists both globally and scoped to `work`, an entity resolving in the `work` category uses the `work` version.
2. **Deeper category beats shallower.** If `priority` exists scoped to both `work` and `work/projects`, an entity in `work/projects` uses the `work/projects` version.
3. **Global fallback.** If no category-scoped definition matches, the global definition applies.

### Multi-Category Scope

A single property can be scoped to **multiple categories simultaneously**. Pass `-c` multiple times at creation (or via `set-scope`):

```bash
rei custom-property create client-code \
  --type-text \
  -e intention \
  -c work/clients -c consulting
```

The property then resolves for entities whose resolution category falls under *any* of the listed scopes.

## Inheritance in the Intention Tree

By default, a property value set on an intention is visible only on that exact intention. Flagging a property definition as **inheritable** changes this: descendants see the nearest ancestor's value, and a descendant can override it by setting its own local value.

### Enabling inheritance

Mark a property inheritable at creation time:

```bash
rei custom-property create focus-area --type-text --inheritable
```

Or toggle it on an existing property:

```bash
rei custom-property set-inheritable focus-area
rei custom-property unset-inheritable focus-area
```

State-machine properties cannot be flagged inheritable (they represent per-entity lifecycle position, not a scalar attribute).

### Observing inheritance

```bash
rei intention set-property --intention ROOT_ID focus-area Product
rei intention show CHILD_ID --full
# ... in the "Custom Properties" section:
#     focus-area: Product (inherited from "Root Intention Title")
```

`rei intention show --json` returns the additional fields `inherited`, `inheritedFromId`, and `inheritedFromTitle` on inherited property rows.

### Overriding and clearing

A local assignment on a descendant shadows the inherited value:

```bash
rei intention set-property --intention CHILD_ID focus-area Research
rei intention show CHILD_ID --full
#     focus-area: Research          (no inheritance marker)
```

Clearing the local value restores the inherited one:

```bash
rei intention clear-property --intention CHILD_ID focus-area
rei intention show CHILD_ID --full
#     focus-area: Product (inherited from "Root Intention Title")
```

### How values are resolved

- Resolution walks up the **intention parent chain** from the target intention.
- The nearest ancestor with a value wins. A local value (depth 0) always wins over any ancestor.
- Only properties flagged inheritable surface ancestor values. Non-inheritable properties show only local values, matching the historical behaviour.
- Matching is by `property_id`, not `property_key`. When category-scoped property keys resolve to different `property_id`s across categories (see [Property Resolution and Inheritance](#property-resolution-and-inheritance)), the descendant only inherits values whose `property_id` matches its own resolved definition — values don't leak across categories.
- Archived or completed ancestors still contribute values (their property assignments are not automatically cleared when they complete; use `clear-property` to remove).

### Filter queries

`rei intention list --where key=value` matches **local** property values only. Inherited values are not matched by the filter. To see which intentions inherit a value, use `rei intention show <id>` on the descendants in question. This limitation may be lifted in a future release.

## Category-Scoped Property Keys

The same property key can exist in different categories with different configurations. This allows category-specific workflows while using consistent property names.

### Creating Properties with Same Key in Different Categories

```bash
# Create a workflow property for work category
rei custom-property create workflow \
  --type-state-machine --from-file work-kanban.yaml \
  -e intention -c work

# Create a workflow property for personal category with different states
rei custom-property create workflow \
  --type-state-machine --from-file personal-kanban.yaml \
  -e intention -c personal
```

### Automatic Resolution

When assigning properties to entities, the system automatically resolves the correct property based on the entity's category:

```bash
# Sets the work workflow property (automatic resolution)
rei intention set-property intention_01abc... workflow in-progress
```

### Disambiguation with --category

When viewing or modifying properties with duplicate keys, use `--category` to specify which one:

```bash
# Show the work version of workflow
rei custom-property show workflow --category work

# Show metrics for personal workflow
rei custom-property metrics workflow --category personal

# Relabel the work version
rei custom-property relabel workflow "Work Pipeline" --category work
```

### Property lookup with --lookup-category

Evolution commands (`add-state`, `remove-state`, `add-transition`, `add-enum-value`, `add-label`, `set-scope`, `init-entities`, `add-state-rule`, `add-condition`, etc.) use `--lookup-category SLUG` for disambiguation instead of `--category`, because `-c` may already have a different meaning on those commands (e.g. `-c` on `set-scope` sets the **new** category scope).

All management and evolution commands also accept a `custom_property_…` **KindID** in place of the key. When a KindID is provided, `--lookup-category` has no effect because the ID is already unambiguous.

```bash
# Disambiguate by category slug
rei custom-property add-state workflow blocked --label "Blocked" \
  --lookup-category work

# Or use the KindID directly (no disambiguation needed)
rei custom-property add-state custom_property_01abc blocked --label "Blocked"
```

### Key Uniqueness Rules

- **Global properties** — key must be unique among all global properties
- **Category-scoped** — key + category combination must be unique
- The same key CAN exist in different categories

## State-Qualified Notes

State machine properties support attaching notes that are qualified by the entity's current state. This lets you maintain per-state documentation (e.g., notes for "in-progress" vs "in-review").

### note

Open or create a state-qualified note for a property's current state on an entity.

```bash
rei custom-property note PROPERTY_KEY [options]
```

**Arguments:**
- `PROPERTY_KEY` — Property key (must be a state machine property)

**Options:**
- `-i, --intention <id>` — Intention ID (uses FZF if not provided)
- `-s, --state <state_id>` — Override state (default: current state from property value)
- `-p, --print` — Print the note path instead of opening editor
- `-t, --tag <tag>` — Tag to add to the note (can be repeated)
- `--actor <actor>` — Actor name
- `-c, --category <slug>` — Category scope for property disambiguation

> **FZF Support:** If `-i` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from available intentions.

**Examples:**

```bash
# Open/create a note for the current state
rei custom-property note workflow -i intention_01abc

# Open a note for a specific state (overrides current)
rei custom-property note workflow -i intention_01abc --state in-review

# Print the note path without opening
rei custom-property note workflow -i intention_01abc --print

# Let FZF pick the intention
rei custom-property note workflow
```

### notes

List state-qualified notes for a property on an entity.

```bash
rei custom-property notes PROPERTY_KEY [options]
```

**Arguments:**
- `PROPERTY_KEY` — Property key (must be a state machine property)

**Options:**
- `-i, --intention <id>` — Intention ID (uses FZF if not provided)
- `-s, --state <state_id>` — Filter to a specific state
- `-c, --category <slug>` — Category scope for property disambiguation

> **FZF Support:** If `-i` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from available intentions.

**Examples:**

```bash
# List all state-qualified notes for an entity
rei custom-property notes workflow -i intention_01abc

# Filter to notes from a specific state
rei custom-property notes workflow -i intention_01abc --state in-progress

# Let FZF pick the intention
rei custom-property notes workflow
```

## Conditional Properties

Conditional properties add value-dependent relevance: a property can only be set when another property has a specific value. This is useful when certain metadata only makes sense in a particular context (e.g., `reading-status` only applies when `content-type = article`).

> **Tip:** `add-condition` and `remove-condition` accept a `custom_property_…` KindID in place of the key, and `--lookup-category SLUG` for disambiguation.

### add-condition

Add a value condition to a property definition.

```bash
rei custom-property add-condition PROPERTY_KEY --when CONDITION
```

**Arguments:**
- `PROPERTY_KEY` — The property to add a condition to

**Options:**
- `-w, --when CONDITION` — Condition expression (see syntax below)

**Condition syntax:**

| Syntax | Operator | Example |
|--------|----------|---------|
| `key=value` | Equals | `--when content-type=article` |
| `key!=value` | Not equals | `--when status!=archived` |
| `key>value` | Greater than | `--when effort>3` |
| `key>=value` | Greater than or equal | `--when priority>=high` |
| `key<value` | Less than | `--when complexity<5` |
| `key<=value` | Less than or equal | `--when due-date<=2026-04-01` |
| `key~v1,v2` | One of | `--when lifecycle~completed,failed` |
| `key?` | Is set (any value) | `--when assigned-to?` |
| `key!` | Is not set | `--when review-notes!` |

**Examples:**

```bash
# reading-status can only be set when content-type = article
rei custom-property add-condition reading-status --when content-type=article

# post-mortem link only relevant when lifecycle is completed or failed
rei custom-property add-condition post-mortem --when lifecycle~completed,failed

# follow-up-date requires assigned-to to be set
rei custom-property add-condition follow-up-date --when assigned-to?
```

### remove-condition

Remove a value condition from a property definition.

```bash
rei custom-property remove-condition PROPERTY_KEY --when CONDITION_KEY
```

**Arguments:**
- `PROPERTY_KEY` — The property to remove the condition from

**Options:**
- `-w, --when CONDITION_KEY` — Property key of the condition to remove

**Example:**

```bash
rei custom-property remove-condition reading-status --when content-type
```

### Condition Semantics

- **Empty list** = unconditional (default, backward compatible)
- **Non-empty list** = ALL conditions must be satisfied (AND logic)
- **Force flag** = `--force` on `set-property` bypasses condition validation
- **Auto-clear** = When a conditioning property's value changes such that a dependent property's condition is no longer met, the dependent property is automatically cleared via a `PropertyClearedByConditionChange` event

### Viewing Conditions

Conditions appear in `property show` output:

```
$ rei custom-property show reading-status
Property: reading-status
  Label: Reading Status
  Type: enum (unread, reading, read, abandoned)

Scope:
  Entity Types: intention
  Category: content

Conditions:
  content-type = article
```

## State Rules

State rules require specific property values before an entity can transition into a state. See the [State Machine Properties Guide](/docs/guides/state-machines) for full documentation.

**Key behavior:** When adding a state rule with a boolean value (e.g., `--require deploy-ready=true`), if the referenced property doesn't exist, it is automatically created as a boolean property inheriting the state machine's scope. This eliminates the need to manually create boolean gate properties.

```bash
# This auto-creates "deploy-ready" as a bool property if it doesn't exist
rei custom-property add-state-rule workflow \
  --state production --require deploy-ready=true
```
