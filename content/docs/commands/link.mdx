Manage external links with canonical URL deduplication. Links are first-class entities with their own lifecycle — the same URL attached to multiple entities shares a single canonical identity.

URLs are automatically normalized using `link-canonical`: tracking parameters are stripped, domain-specific rules applied (YouTube, Amazon, Twitter/X, etc.), and redirect chains resolved. Two different URLs that point to the same resource are recognized as the same link.

## Subcommands

### add

Add a link to an entity. If the canonical URL already exists, the existing link is reused (deduplication). If the link was previously removed, it is auto-restored.

```bash
rei link add <url> [options]
```

**Arguments:**
- `<url>` — The URL to add (required)

**Options:**
- `-i, --intention <id>` — Attach to intention (optional if fzf is available)
- `-a, --action <id>` — Attach to action
- `-o, --outcome <id>` — Attach to outcome
- `-r, --reflection <id>` — Attach to reflection
- `-t, --title <title>` — Link title (optional)
- `--actor <actor>` — Actor that created this link (optional)
- `-p, --property KEY=VALUE` — Set a custom property on the new link (repeatable)
- `--prompt-properties` — After creating the link, interactively walk each custom property applicable to links

> **FZF Support:** If `-i` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select from active intentions.

**Examples:**

```bash
# Add a link to an intention
rei link add https://example.com/docs -i intention_01h455vb4pex

# Let fzf pick the intention interactively
rei link add https://example.com/docs

# Add with a title
rei link add https://github.com/user/repo -i intention_01h455vb4pex -t "Project Repository"

# Add to an action
rei link add https://pr-url.com -i intention_01h455vb4pex -a action_01h455vb4pex

# Same URL added to a different intention shares the canonical link
rei link add https://github.com/user/repo -i intention_01h789abc
```

#### Setting custom properties while adding

You can set custom properties inline when adding a link so you don't have to follow up with `rei link set-property`.

**Non-interactive (scripting / pipelines):** pass one or more `-p KEY=VALUE` flags. Each pair is applied after the link is created; validation errors are reported per-property and do not prevent the link from being created.

```bash
rei link add https://example.com/article \
    -i intention_01h455vb4pex \
    -p status=unread \
    -p priority=high
```

A malformed `-p` argument (e.g. missing `=`) fails at parse time with `expected KEY=VALUE (non-empty KEY, '=' separator)`, and no link is created.

**Interactive walk:** pass `--prompt-properties` to have the CLI walk through every custom property that lists `link` in its allowed entity types after the link is created, prompting for each one. Constrained types (enum, state machine, label set, bool) use an FZF picker; free-form types (text, int, float, date, url, etc.) use a text prompt. Press Esc (or submit empty input) to skip a property. The walk is also automatically skipped when stdin is not a TTY or `fzf` is unavailable, so piped invocations never block on input.

```bash
rei link add --prompt-properties https://example.com/article -i intention_01h455vb4pex
# ... link is created, then:
#   Found 2 custom properties for links. Press Esc on any to skip.
#   status   > (FZF picker)
#   priority > (FZF picker)
```

### list

List links for an entity or across all entities. By default, attachments whose anchor is an archived note are hidden from output (every form: by-anchor, `--all`, `--tree`); pass `--include-archived` to bring them back tagged `[archived parent]`.

```bash
rei link list [options]
```

**Options:**
- `-i, --intention <id>` — Filter by intention
- `-a, --action <id>` — Filter by action
- `-o, --outcome <id>` — Filter by outcome
- `-r, --reflection <id>` — Filter by reflection
- `--all` — Show all links across all anchors
- `-q, --query <text>` — Search links by title or URL (case-insensitive substring match; with `--all`)
- `-d, --domain <domain>` — Filter by registrable domain (e.g., `x.com`, `github.com`; with `--all`)
- `--tree` — Recursively include links from descendant intentions and their notes (with `-i`)
- `--include-removed` — Include removed links (with `--all`)
- `--include-archived` — Include attachments whose parent note is archived. Default behavior excludes them. Each row brought back by this flag is tagged `[archived parent]`. When combined with `--all`, links whose every attachment is archived re-appear in the listing (default behavior excludes such links entirely).
- `--json` — Output as JSON (with `--all`)
- `-w, --where FILTER` — Filter by custom property value (repeatable, AND semantics; see "Filter by custom property" below)

> **FZF Support:** If no anchor is specified and fzf is available, an interactive picker will let you select an entity. Press **Ctrl-F** to toggle between active intentions only and all intentions (including future).

**Examples:**

```bash
# List links for an intention
rei link list -i intention_01h455vb4pex

# List all links across all entities
rei link list --all

# Include removed links
rei link list --all --include-removed

# Filter by domain
rei link list --all --domain github.com

# Search by title or URL
rei link list --all --query "haskell"

# Recursively gather links from an intention and all its descendants
rei link list --tree -i intention_01h455vb4pex
```

After archiving a note that owned a wikilink, that attachment disappears from the default listing and reappears (with the badge) under `--include-archived`:

```text
$ rei link list -i intention_01h455...
# only attachments whose parent notes are active appear

$ rei link list -i intention_01h455... --include-archived
Links for intention_01h455... (3)

  https://en.wikipedia.org/wiki/Foo [archived parent]
    Wikipedia: Foo article
    2026-04-29 14:00
```

#### Filter by custom property

Use `--where` (short: `-w`) to restrict the listing to links whose custom properties match an expression. The flag can be repeated; multiple `--where` flags combine with AND semantics — a link must satisfy every filter to appear in the output. A non-empty `--where` without an anchor or `--tree` is treated like `--all`, so you can query across every link without passing `--all` explicitly.

Supported expression forms (identical to [`rei intention list`](/docs/commands/intention#list)):

- `key=value` — exact match against text, enum state, or integer-as-text.
- `key>=N`, `key>N`, `key<=N`, `key<N` — numeric comparison (integer).
- `key:exists` — the property has any value.
- `key:missing` — the property is not set.
- `key:tag=value` — match a state-machine property by its current state tag.
- `key any v1,v2,...` — label set contains at least one of the listed labels.
- `key all v1,v2,...` — label set contains every listed label.
- `cat/key=value` — qualify by category slug when the same `key` is defined under multiple categories.

**Examples:**

```bash
# Only "to-read" links
rei link list --where status=to-read

# Numeric filter
rei link list --where priority>=3

# Existence / absence checks
rei link list --where topic:exists
rei link list --where topic:missing

# Multiple filters (AND)
rei link list --where status=to-read --where priority>=3

# Label set
rei link list --where tags any haskell,rust
rei link list --where tags all haskell,async

# Category-qualified (when 'priority' is defined in multiple categories)
rei link list --where work/priority=high

# Compose with domain / query / include-removed
rei link list --domain github.com --where topic=haskell
rei link list --include-removed --where topic=haskell

# Anchor + filter
rei link list --intention intention_01h455vb4pex --where topic=haskell
```

If an unqualified `key` matches properties in more than one category, `rei link list` prints a note to stderr suggesting the `cat/key` disambiguation form. Filtering still runs — the note is advisory.

### show

Show link details including canonical URL, status, all attachments, and any custom properties assigned to the link. The `Custom Properties` section is omitted when the link has none.

`rei link show` always lists every attachment of the link, including those whose parent note is archived. Attachments with an archived parent note are tagged `[archived parent]` so you can see why they are missing from default `list` output.

```bash
rei link show [<link-id>]
```

**Arguments:**
- `<link-id>` — The link ID (optional if fzf is available)

> **FZF Support:** If no ID is provided and fzf is available, an interactive picker will let you select a link.

**Options:**
- `--json` — Output as JSON (link row only; custom properties are not included in JSON output)

**Example:**

```bash
rei link show link_01h789abc

# Output:
# Link: link_01h789abc...
# URL: https://www.youtube.com/watch?v=dQw4w9WgXcQ
# Title: Never Gonna Give You Up
# Status: Active
#
# Attachments (2):
#   INTENTION intention_01h455...  "Never Gonna Give You Up"  2026-02-18 10:30
#   INTENTION intention_01h456...  "Rickroll reference"       2026-02-18 10:35
#
# Custom Properties:
#   link-status: watched
#   priority: high
```

### detach

Detach a link from an anchor. The link continues to exist; only the attachment is removed.

```bash
rei link detach [<link-id>] [--attachment <id>]
```

**Arguments:**
- `<link-id>` — The link ID (optional if fzf is available)

**Options:**
- `--attachment <id>` — The attachment ID to detach (required if the link has multiple attachments)

If the link has only one attachment, the `--attachment` flag is not needed.

**Example:**

```bash
rei link detach link_01h789abc --attachment link_att_01h999
```

### title

Set or clear the display title for a link.

```bash
rei link title [<link-id>] [-t <title>] [--clear]
```

**Arguments:**
- `<link-id>` — The link ID (optional if fzf is available)

**Options:**
- `-t, --title <title>` — The new title
- `--clear` — Clear the title

**Examples:**

```bash
# Set a title
rei link title link_01h789abc -t "Project Documentation"

# Clear the title
rei link title link_01h789abc --clear
```

### edit

Correct a link's canonical URL. This is a manual override for when automatic normalization produced an incorrect result.

```bash
rei link edit [<link-id>] --url <new-url>
```

**Arguments:**
- `<link-id>` — The link ID (optional if fzf is available)

**Options:**
- `--url <url>` — The new URL (required)

**Example:**

```bash
rei link edit link_01h789abc --url https://correct-url.com/page
```

### remove

Soft-delete a link. All attachments become inactive. The link can be restored later.

```bash
rei link remove [<link-id>]
```

**Arguments:**
- `<link-id>` — The link ID (optional if fzf is available)

**Example:**

```bash
rei link remove link_01h789abc
```


**Arguments:**
- `<link-id>` — The link ID (optional if fzf is available)

**Example:**


### set-property

Set a custom property on a link. The property must be defined with `rei custom-property create` and must allow the `link` entity type.

```bash
rei link set-property [-l <link-id>] [<property-key>] <value> [--force]
```

**Options:**
- `-l, --link <link-id>` — The link ID (uses FZF if not provided)
- `--force` — Bypass state machine transition rules

**Arguments:**
- `<property-key>` — The property key (uses FZF if not provided)
- `<value>` — The value to set (required)

> **FZF Support:** If `-l` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select a link.

> **FZF Support:** If `<property-key>` is not provided and `fzf` is available, an interactive picker will let you choose from active custom properties that allow the `link` entity type.

**Examples:**

```bash
# Set a read status on a link
rei link set-property -l link_01h789abc read-status reading

# Use FZF to pick the link
rei link set-property read-status reading

# Use FZF to pick the property key
rei link set-property -l link_01h789abc reading
```

> With a single positional argument, optparse-applicative assigns it to `PROPERTY_KEY` and reports that `VALUE` is missing. To drive the property-key FZF picker, supply `VALUE` as the sole positional and either `-l` or FZF-resolve the link.

### clear-property

Clear a custom property from a link.

```bash
rei link clear-property [-l <link-id>] <property-key>
```

**Options:**
- `-l, --link <link-id>` — The link ID (uses FZF if not provided)

**Arguments:**
- `<property-key>` — The property key to clear (required)

> **FZF Support:** If `-l` is not provided and [fzf](https://github.com/junegunn/fzf) is installed, an interactive picker will let you select a link.

**Example:**

```bash
rei link clear-property -l link_01h789abc read-status
```

## Key Concepts

### Canonical URL Deduplication

When you add a URL, it is normalized to a canonical form:
- Tracking parameters (UTM, fbclid, etc.) are stripped
- Domain-specific rules are applied (YouTube, Amazon, Twitter/X, GitHub, etc.)
- Redirect chains are resolved

Two different URLs that resolve to the same resource are recognized as the same link. Adding a URL that already exists reuses the existing link entity.

### One-to-Many Attachments

A single canonical link can be attached to multiple entities. Each attachment can have its own context-specific title. Use `rei link show` to see all attachments for a link.

### Auto-Restore

Adding a URL that matches a previously removed link automatically restores it, preventing accidental duplicates.

## Use Cases

- Attach GitHub PRs or issues to actions
- Link documentation to intentions
- Reference external resources from reflections
- Connect outcomes to published work
- Share the same reference link across multiple related intentions
