This walkthrough follows a realistic software project from start to finish. You'll see how rei's primitives, custom properties, agent delegation, and scheduled automation work together to keep a complex effort organized and moving forward.

## The Project: Building an Event-Sourced API

You want to build a REST API for a side project. It's a meaningful undertaking: multiple components, architectural decisions, and weeks of work. In rei, this isn't a task list. It's a hierarchy of intentions with agents helping at every level.

### Define the Intention Hierarchy

Start with the root intention and break it into meaningful children:

```bash
# The root: why this project matters
rei intention create "Build a portfolio-quality API project" \
  --context personal --horizon 3m

# Children: the major workstreams
rei intention create "Design the API architecture" \
  --parent <root-id> --horizon 1w

rei intention create "Implement authentication" \
  --parent <root-id> --horizon 2w

rei intention create "Build the data layer" \
  --parent <root-id> --horizon 2w

rei intention create "Write API documentation" \
  --parent <root-id> --horizon 1w
```

View the full tree:

```bash
rei intention tree
```

```
Build a portfolio-quality API project [3m]
├── Design the API architecture [1w]
├── Implement authentication [2w]
├── Build the data layer [2w]
└── Write API documentation [1w]
```

Each child intention has its own actions, outcomes, blockers, and delegation. The parent holds the big picture. Notice there's no "deploy" intention: deployment is a phase in the workflow state machine, not a separate workstream.

### Add Custom Properties for Shared Context

Custom properties give both you and your agents structured context about each intention. Define them once, use them everywhere:

```bash
# Project path: tells agents where the code lives
rei custom-property create project-path \
  --type-path --must-exist --must-be-directory \
  -e intention

# Repository URL: agents can reference PRs and issues
rei custom-property create repo-url \
  --type-url --allowed-schemes https \
  -e intention

# Workflow state machine: tracks each intention through a dev lifecycle
rei custom-property create workflow \
  --type-state-machine --from-file dev-workflow.yaml \
  -e intention
```

The workflow state machine (`dev-workflow.yaml`) defines your full development process, from backlog through deployment:

```yaml
initialState: backlog
states:
  - stateId: backlog
    stateLabel: "Backlog"
    stateOrder: 1
    stateTags:
      - systemTag: TagQueue
  - stateId: in-progress
    stateLabel: "In Progress"
    stateOrder: 2
    stateLimit: 3
    stateTags:
      - systemTag: TagActive
  - stateId: in-review
    stateLabel: "In Review"
    stateOrder: 3
    stateTags:
      - systemTag: TagActive
      - userTag: "needs-review"
  - stateId: deploying
    stateLabel: "Deploying"
    stateOrder: 4
    stateTags:
      - systemTag: TagActive
  - stateId: done
    stateLabel: "Done"
    stateOrder: 5
    stateTags:
      - systemTag: TagDone
      - systemTag: TagTerminal
transitions:
  - transitionFrom: backlog
    transitionTo: in-progress
    transitionLabel: "Start"
  - transitionFrom: in-progress
    transitionTo: in-review
    transitionLabel: "Submit for review"
  - transitionFrom: in-review
    transitionTo: in-progress
    transitionLabel: "Request changes"
  - transitionFrom: in-review
    transitionTo: deploying
    transitionLabel: "Approve for deploy"
  - transitionFrom: deploying
    transitionTo: done
    transitionLabel: "Deployed"
  - transitionFrom: deploying
    transitionTo: in-progress
    transitionLabel: "Rollback"
```

The WIP limit of 3 on "In Progress" keeps you from spreading too thin. Deployment is a workflow phase, so every intention flows through the same lifecycle. Now assign properties to your intentions:

```bash
# Set the project path and repo on the root intention
rei intention set-property <root-id> project-path ~/projects/my-api
rei intention set-property <root-id> repo-url https://github.com/you/my-api

# Move the first workstream to in-progress
rei intention set-property <design-id> workflow in-progress
```

When any agent coaches you, reviews your work, or executes a delegation, it sees these properties. It knows where the code lives, what repo to reference, and what workflow phase each intention is in.

### Create a Project Dashboard View

Views let you check on the project at a glance. Create a `project-board.yaml`:

```yaml
sections:
  - label: "Kanban Board"
    target: children
    filters:
      - type: status
        value: active
    groupBy: workflow
    showProperties: [workflow]

  - label: "Blocked"
    target: descendants
    filters:
      - type: state_tag
        property: workflow
        tag: blocked
    hideIfEmpty: true

  - label: "Recent Actions"
    target: actions
    limit: 5

  - label: "Project Notes"
    target: notes
    limit: 5
    hideIfEmpty: true

showDormant: false
```

Register and run it:

```bash
rei view create project-board --from-file project-board.yaml -d "API project kanban"
rei view exec project-board --scope <root-id>
```

```
project-board
  API project kanban

━━━ Kanban Board (4) ━━━
  ▸ In Progress (2)
    ● Implement authentication
    ● Build the data layer
  ▸ In Review (1)
    ● Design the API architecture
  ▸ Backlog (1)
    ● Write API documentation

━━━ Recent Actions (3) ━━━
  2026-03-25  Set up JWT middleware with refresh token rotation
  2026-03-24  Chose MessageDB based on agent research

```

Run this view any time with `rei view exec project-board --scope <root-id>`, or use `rei intention view` with FZF to pick the intention and view interactively.

### Add Guidance for Agent Interactions

Guidance is persistent context that shapes how agents approach an intention. It's included in every agent interaction, whether that's coaching, reviewing, assisting, or delegated work. Edit it in your `$EDITOR`:

```bash
rei intention guidance <design-id>
```

This opens a markdown file where you describe what matters for this intention. For example:

```markdown
## Dependency Lookup
Always use `mori` to find preferred libraries and patterns before suggesting
alternatives or guessing at APIs.

- `mori registry search <name>` to find registered packages
- `mori registry show <project> --full` for source paths and metadata
- `mori registry docs <project>` for guides, cookbooks, and references
- `mori deps locate <dep>` to resolve a dependency to its local path

Read the source code and docs on disk to understand APIs and behavior.

## Project Context
This is a Haskell project using GHC 9.12 with Effectful for effects,
MessageDB for event sourcing, and optparse-applicative for CLI.
Use `mori registry show rei --full` to see the full dependency graph.

## When coaching
Focus on whether the architecture supports the long-term vision, not just
immediate needs. Challenge scope decisions, not implementation details.
```

This guidance is loaded into every agent session that touches this intention. Unlike delegation instructions (which are task-specific), guidance persists and informs all future agent interactions.

### Delegate the Architecture Research

The architecture design needs research. Instead of spending a day evaluating options yourself, delegate it with specific instructions:

```bash
rei intention delegate <design-id> \
  --to claude-code \
  --scope research \
  --autonomy supervised \
  --instructions "Evaluate MessageDB vs EventStoreDB vs custom event store. Compare on: Haskell client support, subscription model, operational complexity, and PostgreSQL integration."
```

The agent receives the delegation instructions, plus the intention's persistent guidance, the project path, the repo URL, and the parent context. It investigates and submits findings for your review.

### Review and Act on Agent Research

The agent finishes and submits a review checkpoint. You see it in your dashboard:

```bash
rei today
```

```
Active Cycle: Discipline Cycle 3 (Day 4/10)
Focus: Create

Active Intentions (In Progress)
  ⚡ Design the API architecture [1w]
      workflow: in-progress | delegated to: claude-code (research)

Pending Reviews
  ✎ Research: "Design the API architecture"
    Agent: claude-code | Scope: research | Submitted: 1h ago
    Artifact: note_01abc... (Architecture evaluation)

Habits Due Today
  □ Daily coding practice (flexible)

Agent Schedules
  ⏰ Morning standup (daily at 07:00) | last run: today 07:00
```

Read the agent's research note, then decide:

```bash
# Approve the research
rei review approve <review-id>

# Or send it back with feedback
rei review request-changes <review-id> \
  --feedback "Good overview, but dig deeper into MessageDB's subscription model"
```

Once approved, record your decision as an action:

```bash
rei action record <design-id> "Chose MessageDB based on agent research. PostgreSQL-native, simple subscription model."
```

Move the intention forward in the workflow:

```bash
rei intention set-property <design-id> workflow in-review
```

### Set Up Scheduled Agents

You want agents working in the background without manual invocation:

```bash
# Morning standup: daily review of active intentions
rei agent-schedule add \
  --trigger daily-at:07:00 \
  --scope children-of:<root-id> \
  --guidance "Review progress on active intentions. Flag anything stalled for more than 2 days. Suggest what to focus on today based on workflow states and blockers."

# Auto-research blockers: when you declare a blocker, an agent investigates
rei agent-schedule add \
  --trigger on-blocker-declared \
  --scope children-of:<root-id> \
  --guidance "Research the blocker using the project source code at the project-path property. Suggest concrete approaches to resolve it."

# Stall detection: catch intentions that go quiet
rei agent-schedule add \
  --trigger on-stalled:3d \
  --scope children-of:<root-id> \
  --guidance "This intention has had no activity for 3 days. Check if it's blocked, deprioritized, or forgotten. Suggest next steps."
```

These agents run autonomously. Each execution creates a session with timing, tokens, and a summary. You see the results in `rei today` and can inspect any session:

```bash
rei agent sessions
rei agent session show <session-id>
```

### Work Through the Implementation

As you build, the system captures everything. Start the auth implementation:

```bash
rei intention set-property <auth-id> workflow in-progress

rei action record <auth-id> "Set up JWT middleware with refresh token rotation"
rei action record <auth-id> "Added integration tests for auth flows" --duration 2h
```

Hit a blocker:

```bash
rei blocker declare <auth-id> "OAuth provider documentation is unclear on PKCE flow edge cases"
```

Because you set up the `on-blocker-declared` schedule, an agent automatically picks this up. It reads the project source code (using the `project-path` property), investigates the PKCE flow, and posts findings as an action on the intention. You see it next time you check your dashboard.

### Delegate Documentation Drafting

Writing docs is valuable but time-consuming. Delegate it with specific instructions:

```bash
rei intention delegate <docs-id> \
  --to claude-code \
  --scope draft \
  --autonomy supervised \
  --instructions "Generate OpenAPI spec and a getting-started guide. Reference the actual endpoint implementations at the project-path. Follow the conventions in the existing README."
```

With `draft` scope, the agent creates artifacts (notes and files) for your review. With `supervised` autonomy, it submits a review checkpoint before anything is finalized. The agent also sees any guidance you've set on this intention, giving it persistent context about your preferences. You review the draft, request changes if needed, and approve when it's ready.

### Mark Dependencies Between Intentions

Documentation depends on the API being built. Make that explicit:

```bash
rei intention add-dependency <docs-id> --on <auth-id>
rei intention add-dependency <docs-id> --on <data-id>
```

When you check `rei today`, the dashboard shows that the docs intention is waiting on two dependencies. The morning standup agent sees this too and won't suggest working on docs until the dependencies are resolved.

### Track Progress with Metrics and Views

Check your project board to see the current state:

```bash
rei view exec project-board --scope <root-id>
```

Or get the raw workflow metrics:

```bash
rei custom-property metrics workflow
```

```
State Distribution
  in-progress:  1  (Build the data layer)
  in-review:    1  (Implement authentication)
  deploying:    1  (Design the API architecture)
  done:         1  (Write API documentation)
```

Record outcomes as milestones are reached:

```bash
rei outcome record <auth-id> "Auth system passes all integration tests, ready for review"
rei intention set-property <auth-id> workflow in-review
```

### Reflect on the Week

At the end of the week, reflect with agent assistance:

```bash
rei agent reflect week
```

The agent reviews your week's actions across all child intentions, the agent delegations and their results, blockers resolved and still open, and workflow state changes. It asks questions that help you think about the bigger picture:

> *You moved three intentions forward this week and resolved two blockers. The auth implementation took longer than your 2-week horizon suggested. The agent's PKCE research saved you time, but you spent a full day on refresh token edge cases. Is the 2-week horizon still realistic, or should you adjust? The data layer hasn't started yet and depends on the auth design. What's the risk of that becoming the critical path?*

Your reflection is recorded and informs future coaching sessions.

### Complete the Project

As intentions reach their terminal workflow state, they can auto-complete:

```bash
# Designate workflow as a lifecycle property with 3-day auto-complete
rei custom-property set-lifecycle workflow --days 3
```

When an intention moves through review, deployment, and finally to `done`, rei schedules auto-completion in 3 days. If you realize there's more work, just move it back to an earlier state and the auto-complete is cancelled.

Once all children are complete, record the final outcome on the root:

```bash
rei outcome record <root-id> "API deployed, documented, and serving production traffic"
rei intention complete <root-id>
```

## What This Walkthrough Covered

| Capability | How it was used |
|-----------|----------------|
| **Intention hierarchy** | Root intention with four child workstreams, each with its own lifecycle |
| **Custom properties** | `project-path`, `repo-url`, and a `workflow` state machine with WIP limits and a deploying phase |
| **Intention views** | Kanban board view grouped by workflow state for at-a-glance project status |
| **Intention guidance** | Persistent agent context: coaching style, research preferences, constraints |
| **Agent delegation** | Research (architecture evaluation), draft (API documentation), with task-specific `--instructions` and supervised review |
| **Review checkpoints** | Structured approve/request-changes workflow for agent-produced artifacts |
| **Scheduled automation** | Morning standup, auto-research on blockers, stall detection |
| **Dependencies** | Documentation blocked on auth and data layer completion |
| **Workflow metrics** | State distribution across all project intentions |
| **Lifecycle automation** | Auto-complete intentions when they reach a terminal workflow state |
| **Agent-assisted reflection** | Weekly review that synthesizes progress across the whole project |

This is what agent-augmented intentional living looks like for a real project: you focus on the creative and decision-making work, agents handle research, drafting, and background monitoring, and the system keeps everything transparent, attributed, and traceable.
