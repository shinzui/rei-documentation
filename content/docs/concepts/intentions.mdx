Intentions are the core building blocks of rei. Unlike tasks, intentions represent meaningful commitments to change something in the world. They're also the unit of collaboration between you and your agents: you can delegate intentions, attach guidance, and review agent work at the intention level.

## What is an Intention?

An intention is not just a task; it's a commitment to a change you want to see happen. Tasks are disposable and interchangeable; intentions have identity and meaning. This distinction matters because agents that understand your *intentions* give better coaching, more relevant research, and more focused execution than agents working from a task list.

**Examples:**
- "Learn to play piano" (not "Practice piano for 30 minutes")
- "Build a personal website" (not "Write HTML files")
- "Become a better writer" (not "Write 500 words")

## Intention Hierarchy

Intentions can have parent-child relationships, allowing you to organize work by meaning:

```
Connect with like-minded people (parent)
└── Become a better writer (child)
    └── Publish first blog post (grandchild)
```

The parent provides context. When you're deep in the details, the parent reminds you *why* it matters.

## Status Lifecycle

Intentions have one of four statuses:

| Status | Description |
|--------|-------------|
| **Active** | Currently being worked on. Shown in daily dashboard. |
| **Future** | Planned but not yet active. Hidden until activated. |
| **Completed** | Successfully finished. Terminal state. |
| **Abandoned** | No longer pursuing. Terminal state with optional reason. |

### Lifecycle Flow

```
           ┌─────────────────────┐
           │                     │
   create  ▼      defer          │  activate
    ───► Active ◄────────► Future ◄────┘
           │
           │ complete/abandon
           ▼
       Terminal
```

- New intentions start as **Active** by default
- Use `defer` to move to **Future** (also defers children)
- Use `activate` to move back to **Active**
- Use `complete` or `abandon` for terminal states

## Context

Context separates different life domains (e.g., work vs personal):

```bash
rei intention create "Quarterly planning" --context work
rei intention create "Learn Korean" --context personal
```

Filter your view by context:

```bash
rei today --context work
```

<Callout type="info">
Context can only be set on root intentions. Children inherit context from their root ancestor.
</Callout>

## Horizon

Horizon provides a time frame estimate:

```bash
rei intention create "Learn Haskell" --horizon 3m
rei intention create "Fix bug" --horizon 1d
```

Common horizons:
- `1d` (one day)
- `1w` (one week)
- `1m` (one month)
- `1y` (one year)
- Custom text like `"Q1 2024"`

## Agent Integration

Intentions are the primary unit of human-agent collaboration:

- **Guidance**: persistent context (opened in `$EDITOR`) that shapes how agents approach this intention across all interactions
- **Delegation**: assign an intention to an agent with explicit scope (research, draft, execute), autonomy level, and task-specific `--instructions`
- **Review**: agents submit work for your approval through structured review checkpoints
- **Timeline**: `rei intention timeline <id>` shows interleaved human and agent actions

```bash
# Edit persistent guidance for this intention (opens $EDITOR)
rei intention guidance <id>

# Delegate with task-specific instructions
rei intention delegate <id> --to claude-code --scope research \
  --instructions "Evaluate library options for OAuth2 PKCE flow"
```

## Related Concepts

Intentions connect to other primitives:

- **Actions**: record work done toward an intention (by you or agents)
- **Outcomes**: record evidence of progress
- **Blockers**: track what's preventing progress (can trigger agent schedules)
- **Dependencies**: mark intentions that depend on others
- **Habits**: link recurring practices to intentions
